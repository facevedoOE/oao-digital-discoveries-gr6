Flowcharts & Algorithm Design

Pseudocode is great — but sometimes a picture is faster than words. This week, you'll learn the visual language of flowcharts: five simple shapes that can sketch any algorithm in the world.

Learning Objectives

By the end of this week, you will be able to:

1
Identify the five core flowchart shapes — terminator, input/output, process, decision, and arrow — and explain what each one means.
2
Read a flowchart by following the arrows from start to end, including branches at decision diamonds.
3
Draw your own flowchart for a real-world routine, choosing the right shape for each step.
4
Translate between flowchart, pseudocode, and code — they all describe the same algorithm in different forms.
01
Section 1 · From Pseudocode to Pictures

Why a flowchart, when you have words?

Last week, you learned to sketch algorithms in pseudocode — English-flavored steps that humans can read at a glance. Pseudocode is a fantastic tool. But the moment your algorithm needs to branch ("if this, do A; otherwise, do B"), pseudocode starts to get tangled. The reader has to track indentation, follow nested IFs, and hold the whole shape in their head.

That's where flowcharts shine. A flowchart is the same algorithm, drawn as a picture: shapes for the steps, arrows for the flow, diamonds for the decisions. The branching that was hidden inside indented text is now visible at a glance.

Programmers, doctors, factory engineers, and even chefs use flowcharts every day to plan, share, and debug processes. By the end of this week, you'll be drawing them too — and reading anyone else's like a map.

A simple flowchart: "Is the number even?" — start at the top, follow the arrows.

📘 Vocabulary

Flowchart — a picture that shows the steps of an algorithm using shapes connected by arrows.

Branch — a place in a flowchart (or any algorithm) where the path splits, usually because of a decision.

Sequence — the order in which steps run. Flowchart arrows show the sequence visually.

02
Section 2 · The Five Core Shapes

Five shapes can sketch any algorithm

Flowcharts use a tiny vocabulary — just five shapes — and that's all you need to describe any program in the world. Each shape has a specific job:

⬭Terminator (Oval)
Says: "Start" or "End." Every flowchart begins and ends with one. Example: START, END.
▱Input/Output (Parallelogram)
Says: "Get something from the user" or "Show the user something." Example: Get N, Print "Even".
▭Process (Rectangle)
Says: "Do a step" — a calculation, an assignment, an action. Example: SET total = a + b.
◇Decision (Diamond)
Says: "Ask a yes/no question." Has TWO arrows leaving — one labeled YES, one labeled NO. Example: Is N > 10?
→Arrow (Connector)
Says: "Go this way next." Arrows connect every shape and show the order of the steps.
💭 Think About It

The diamond is the only shape that has two arrows leaving. That's what makes it special — every other shape just leads to the next thing, but a diamond splits the path based on a question. Diamonds are where decisions happen.

Section 3 · Reading a Flowchart

Follow the arrows, top to bottom

Reading a flowchart is the easiest part. You start at the terminator at the top, follow each arrow, do whatever the shape says, and stop when you hit the terminator at the bottom. Decisions split the path — pick the YES or NO arrow based on the answer.

Here's a flowchart for a question every Floridian asks at least once a week: "Should I bring an umbrella?"

Walk through it: you START → check the weather (input) → reach the diamond and ask "is rain in the forecast?" → if YES you bring the umbrella, if NO you leave it home → both paths merge → END. The whole algorithm fits in your head at a glance, even though it has a branch.

Notice all five shapes are here: a green oval (start), a blue parallelogram (input), a gold diamond (decision), two purple rectangles (processes), and a final purple oval (end). That's the entire flowchart vocabulary.

✅ Check for Understanding

Pause and trace: In the umbrella flowchart above, suppose the forecast says "60% chance of rain." Which path would you take, and which two shapes would you visit before reaching END?

04
Section 4 · Drawing Your Own

Four rules for clean flowcharts

Once you can read flowcharts, drawing them is just practice. Follow these four rules and your charts will be readable by anyone — including future you, three weeks from now when you've forgotten what you wrote.

1
Always start and end with a terminator (oval)
No exceptions. Every flowchart needs at least one START and one END. Some flowcharts have multiple ENDs if there are different ways out.
2
Use diamonds only for yes/no questions
A diamond always has two arrows leaving — labeled YES and NO. If your "decision" doesn't have two clean answers, it's not really a decision; it's a process step.
3
Arrows go in one direction
Top-to-bottom is standard. Crossing arrows or arrows pointing backwards make charts hard to follow. (We'll relax this for loops in Week 24!)
4
Keep each shape short — one verb, one noun
"Add the totals," "Print the answer," "Get user's age." If a shape needs three sentences, break it into more shapes.
🌎 Real-World Connection

Hospitals use flowcharts called clinical pathways to standardize emergency treatment. When a patient arrives with chest pain, a nurse follows a flowchart with decision diamonds — "Is the pain on the left side?", "Is the patient over 50?", "Is the EKG abnormal?" — that routes the patient to the right care, fast.

The same kind of branching diagram you're learning to draw is saving lives in emergency rooms right now.

Section 5 · Three Forms, One Algorithm

Flowchart → Pseudocode → Code

A flowchart, a piece of pseudocode, and a working program are all just different ways to express the same algorithm. Once you can move between them fluently, you can sketch on a whiteboard and translate it to code in minutes.

Here's the "Is N even?" algorithm in all three forms:

📊 Flowchart
A picture
  • ⬭ START
  • ↓
  • ▱ Get N
  • ↓
  • ◇ N % 2 == 0?
  • YES → "Even"
  • NO → "Odd"
  • ↓
  • ⬭ END
📝 Pseudocode
Plain-English steps
  • GET N
  • IF N % 2 == 0
  •   PRINT "Even"
  • ELSE
  •   PRINT "Odd"
🐍 Python
Real code (Week 21!)
  • n = int(input())
  • if n % 2 == 0:
  •   print("Even")
  • else:
  •   print("Odd")

Same algorithm. Three different forms. The flowchart is best for sharing with non-coders. The pseudocode is best for thinking. The Python is best for actually running on a computer. Mastering the translation between them is one of the most useful skills a programmer has.

💡 Did You Know?

The flowchart shapes you're using were standardized by the American National Standards Institute (ANSI) in 1970 — that's the same year the first floppy disk was released. The five shapes have stayed the same for over fifty years because they work. When you draw a flowchart in 2026, you're using a visual language that engineers have been sharing across the planet for half a century.

📚 Key Vocabulary — Week 20

Flowchart
A picture that shows the steps of an algorithm using shapes connected by arrows.
Terminator
An oval shape that marks the start or end of a flowchart. Every flowchart begins and ends with one.
Input/Output
A parallelogram shape used for getting information from the user or showing information back to them.
Process
A rectangle shape used for "do a step" — a calculation, an assignment, or any normal action.
Decision
A diamond shape that asks a yes/no question and splits the path into two branches.
Branch
A point in a flowchart (or any algorithm) where the path splits, usually because of a decision.
Sequence
The order in which steps run. In a flowchart, the arrows show the sequence.
Connector arrow
An arrow pointing from one shape to the next, showing where the flow goes.
