───────────────────────────────────────────────────────────────
Q1. Which of these is the BEST definition of decomposition in
    computer science?
───────────────────────────────────────────────────────────────
  A) Removing parts of a problem you don't want to solve
     ❌ Decomposition doesn't skip parts — it breaks the WHOLE
        problem into smaller pieces so every part still gets
        solved.
  B) Breaking a big problem into smaller, easier sub-problems
     you can solve one at a time                              ✓ CORRECT
     ✅ Decomposition takes one giant task and splits it into
        pieces small enough that you (or a computer) can
        actually solve each one.
        (SC.K12.CTR.1.1)
  C) Writing the answer to a problem before you understand it
     ❌ That's guessing. Decomposition happens BEFORE you solve
        anything — it's the planning step where you organize
        the problem.
  D) A type of computer error that crashes your program
     ❌ Decomposition isn't an error — it's a problem-solving
        strategy. Every programmer uses it on every project.

───────────────────────────────────────────────────────────────
Q2. Carlos is decomposing the big task "Plan a class field trip."
    Which of these is the BEST sub-problem from his decomposition?
───────────────────────────────────────────────────────────────
  A) "Plan a class field trip"
     ❌ That's the WHOLE big problem, not a sub-problem. A
        sub-problem is a smaller piece you'd get after
        decomposing.
  B) "Make everyone happy"
     ❌ "Make everyone happy" is a feeling, not a concrete
        task. Sub-problems should be specific, doable actions.
  C) "Pick a date when the buses are available"             ✓ CORRECT
     ✅ That's a small, specific sub-problem with a clear
        answer (a date). You could solve this one in a few
        minutes by checking the bus schedule.
        (SC.K12.CTR.1.1)
  D) "Be a great class president"
     ❌ That's a value or identity goal, not a problem to
        solve. Sub-problems are concrete tasks with a clear
        finish.

───────────────────────────────────────────────────────────────
Q3. When should you STOP decomposing a sub-problem and start
    solving it directly?
───────────────────────────────────────────────────────────────
  A) When the sub-problem has fewer than 5 letters in its name
     ❌ The length of the name doesn't tell you anything about
        the size of the problem. Stop based on whether you can
        SOLVE it directly.
  B) When the sub-problem is small enough that you could write
     a clear algorithm for it without breaking it down again ✓ CORRECT
     ✅ That's the test. If a piece is small enough to write
        5–10 clear algorithm steps for, it's ready to solve.
        No need to decompose further.
        (SC.K12.CTR.1.1)
  C) You should never stop — keep decomposing forever
     ❌ Decomposing forever means you never solve anything.
        There's a clear stopping point: when each piece is
        small enough to solve directly.
  D) When you get tired of thinking about it
     ❌ Tiredness isn't the test. Pieces that are still too big
        will trip you up later — push through and break them
        down further.

───────────────────────────────────────────────────────────────
Q4. Why is decomposition especially important when many
    programmers work together on a huge project (Minecraft,
    Google, NASA's moon mission)?
───────────────────────────────────────────────────────────────
  A) It makes the project slower on purpose
     ❌ Decomposition makes huge projects FASTER, not slower —
        work happens in parallel and bugs are easier to find.
  B) Decomposition diagrams look impressive in presentations
     ❌ Diagrams do look cool, but the real reason is
        functional. Without decomposition the work would be
        impossible to coordinate.
  C) It splits the work into pieces so different people can
     work on different sub-problems at the same time        ✓ CORRECT
     ✅ The navigation team works on navigation while the
        life-support team works on life support — in parallel.
        That's only possible because the big mission was
        decomposed into independent sub-problems first.
        (SC.K12.CTR.2.1)
  D) It removes the need to write any code at all
     ❌ Decomposition organizes the code that gets written —
        it doesn't replace it. You still have to solve every
        sub-problem.

───────────────────────────────────────────────────────────────
Q5. How do decomposition and algorithms work together when you
    build software?
───────────────────────────────────────────────────────────────
  A) They are the same thing — just two different words for it
     ❌ They're related but different. Decomposition organizes
        the problem; algorithms solve each piece.
  B) You only need decomposition — algorithms are optional
     ❌ Decomposition gives you the pieces, but each piece
        still needs a clear algorithm or nothing actually
        gets done.
  C) Decomposition decides WHAT to solve. Algorithms decide
     HOW to solve each piece.                                ✓ CORRECT
     ✅ First decompose the big problem into small ones (the
        WHAT), then write a clear algorithm for each small
        piece (the HOW). Together they turn impossible
        problems into doable ones.
        (SC.K12.CTR.1.1, CTR.2.1)
  D) Algorithms are for computers, decomposition is for
     humans only
     ❌ Both are for humans AND computers. Decomposition
        organizes the problem for whoever or whatever will
        solve it.

  TEACHER NOTES
═══════════════════════════════════════════════════════════════

This is the SECOND week of Unit 3 (Programming) — the
decomposition pillar of computational thinking, building
directly on Week 19's introduction to algorithms.

Students who struggle with this quiz often need help with:
  • Distinguishing the BIG problem from a SUB-problem (Q2).
    Drill this with a quick game: shout a big task ("clean
    your room"), have students name 3 sub-problems out loud.
  • Knowing when to STOP decomposing (Q3). Use the rule of
    thumb: "Could you write a clear 5-10 step algorithm for it?
    If yes, stop. If no, decompose again."
  • The connection between decomposition and algorithms (Q5).
    Walk through the PB&J algorithm from Week 19 — point out
    that "Make a sandwich" was already a decomposition
    (bread + PB + knife + spread + stack) before it became an
    algorithm.

The reflection assignment (decompose your weekend) and the VR
Decomposition Tree Builder are the best assessments of mastery
this week. Look for:
  • A clearly stated big problem at the top
  • 4-6 sub-problems on the first level
  • At least one sub-problem decomposed to a second level
  • No vague labels like "make it good" or "do the rest"

NEXT WEEK (Week 21): Pattern Recognition. Students will start
spotting WHICH of their decomposed pieces are secretly the same
across different problems — the third pillar.
