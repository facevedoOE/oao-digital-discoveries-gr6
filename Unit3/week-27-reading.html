01
Section 1 · The Big Picture

From Idea to Algorithm to Code

Every programmer — your favorite app developer, the person who built the game you played last night, you — follows the same arc. It starts with a fuzzy idea in someone's head and ends with running code on a screen. The trip in between is what this whole unit was about.

The arc has three big stages. You learned them as separate weeks, but they're really one continuous motion:

Stage 1 — Idea — is messy. "I want a program that quizzes me on Spanish vocab." That sentence is too vague for a computer. So you use computational thinking to break it apart.

Stage 2 — Algorithm — is where the idea becomes concrete: ordered steps, written in pseudocode or drawn as a flowchart. Anyone (even a robot) could follow your algorithm and get the same result. Real programmers spend a huge amount of time here. The cleaner this stage, the easier the next one is.

Stage 3 — Code — is where you translate the algorithm into a real programming language like Python. If your algorithm is solid, this is mostly translation. If your algorithm is shaky, this is where you discover that and have to go back. (That round-trip is normal — we'll cover it in Section 5.)

🤔 Think About It

Pick something you do most days — like brushing your teeth, packing your lunch, or starting a video game. Right now, that's a Stage 1 idea sitting in your head. What would Stage 2 look like for it? Roughly how many steps would the algorithm have? Where would the first decision happen?

Section 2 · Computational Thinking + Flowcharts

Stage 1 → Stage 2: turning a fuzzy idea into a clear plan

Way back in Week 19 you met the four pillars of computational thinking. They are the four moves your brain makes to turn "I want a program that…" into "here are the steps." Quick refresher:

PILLAR 1 · 🧩
Decomposition
Break a big problem into smaller pieces you can solve one at a time. "Build a quiz app" → guest list of features: questions, answers, scoring, results screen.
PILLAR 2 · 🔁
Pattern Recognition
Notice when two pieces look the same. "Show one question and check the answer" repeats for every question — that's a pattern, and patterns become loops.
PILLAR 3 · 🔍
Abstraction
Keep what matters, ignore what doesn't. The color of the question text doesn't matter for grading — so don't worry about it yet. Focus on the logic.
PILLAR 4 · 📋
Algorithm Design
Write the actual ordered steps. This is what a flowchart and pseudocode capture. By now the problem is small enough that the steps almost write themselves.

In Week 20 you learned how to draw those steps as a flowchart. Flowcharts use four simple shapes — and once you know them, you can read or draw a flowchart for any algorithm:

SHAPES THAT GO STRAIGHT THROUGH
Oval, Rectangle, Parallelogram
  • Oval — START or END. Every flowchart has one of each.
  • Rectangle — a process step. "Add 1 to score." One arrow in, one arrow out.
  • Parallelogram — input or output. "Get age from user." "Print the result."
SHAPE THAT SPLITS THE PATH
Diamond
  • Diamond — a decision (a yes/no question).
  • Always has one arrow in and two labeled arrows out (Yes/No or True/False).
  • Diamonds become if statements when you turn the flowchart into Python.
🌎 Real-World Connection

Flowcharts aren't just for programmers. Doctors, pilots, and emergency dispatchers use them every day. When a 911 dispatcher answers a call, they're literally walking down a flowchart: "Is anyone hurt? Yes → ambulance branch. No → next question." The shapes you learned in Week 20 are part of how the world stays organized.

✅ Quick Check

You're flowcharting "Buy a snack from the vending machine." Which shape would you use for each of these steps?

  1. "Begin" at the top of the chart
  2. "Insert dollar bill"
  3. "Is enough money inserted?"
  4. "Print 'Enjoy your snack!'"

Answers: 1 — oval, 2 — parallelogram (input), 3 — diamond, 4 — parallelogram (output).

Section 3 · Python Basics + Logic

Stage 3, Part 1: variables, types, and IF/ELIF/ELSE

Once your algorithm is solid, you translate it into Python. The most important things you learned about Python live in two weeks: Week 21 (variables and types) and Week 23 (conditionals).

A variable is a labeled box that holds a value. The label is whatever you choose; the box's type tells Python what KIND of thing fits inside it. Types matter — they decide what operations make sense. You can add 5 + 3, but you can't add "hello" + 3; one is a number and one is text.

TYPE
EXAMPLE
USE IT WHEN…
int
age = 12
…you have a whole number — counts, scores, ages, level numbers.
float
price = 4.99
…you need a decimal — money, averages, weights, distances.
str
name = "Maya"
…you have text — anything in quotes, even "100" (a string that LOOKS like a number).
bool
is_open = True
…you only need True or False — flags, yes/no answers, on/off switches.
💡 Did You Know?

The bool type is named after George Boole, an English mathematician from the 1800s. He wrote a book called The Laws of Thought that worked out the math of TRUE and FALSE — almost a hundred years before anyone built a computer. Every if statement you write traces back to him.

In Week 23 you learned how to make your code choose between paths using if, elif, and else. Here's the rule that trips students up most often: in an if/elif/else chain, only ONE branch ever runs. Python checks them top to bottom and stops at the first True. The rest are skipped.

▶ Conditional review (Week 23) # Tells you which lunch line to use based on grade grade = 6 if grade <= 5: print("Elementary line") elif grade <= 8: print("Middle school line") else: print("High school line")

For grade = 6: the first check (grade <= 5) is False, so we skip to the elif. 6 <= 8 is True — so we print "Middle school line" and the else never runs. If grade had been 11, BOTH the if and elif would fail and the else would catch it.

One more thing about logic: notice the difference between = (one equals — assignment, "put this in the box") and == (two equals — comparison, "are these the same?"). Mixing them up is one of the most common bugs in any programmer's life.

📘 Quick Vocab Recap

Assignment (=) — puts a value into a variable. Always changes something.

Comparison (==, !=, >, <, >=, <=) — asks a question. Always returns a bool (True or False), never changes anything.

Indentation — in Python, the spaces at the start of a line tell Python what code BELONGS to which block. Lining things up isn't decoration; it's structure.

Section 4 · Loops + Functions

Stage 3, Part 2: making code repeat and reusing it

By the time you reached Week 24, you'd noticed something: a lot of code wants to repeat. Print every score in a list. Ask the user three times. Loop a game until they quit. So you learned loops — a way to write a block of code once and tell Python "do that again."

You met two kinds: for loops (when you know how many times to repeat) and while loops (when you repeat until something happens). The for loop with range() is the workhorse of beginner Python:

▶ for loop with range (Week 24) for i in range(1, 4): print("Round", i) # Prints: Round 1 Round 2 Round 3 # range(1, 4) gives 1, 2, 3 — STOPS BEFORE 4.

Then in Week 25 you met functions. A function is a named, reusable block of code. You define it once with def; then you can call it as many times as you want — with different inputs each time. The big idea: defining a function does NOT run the code inside it. It's like saving a recipe in a cookbook. The recipe doesn't cook anything until somebody actually opens the book and uses it.

▶ Function definition + call (Week 25) def greet(name): # DEFINE — doesn't run yet return "Hello, " + name # Now CALL it three times with different inputs: print(greet("Maya")) # Hello, Maya print(greet("Sam")) # Hello, Sam print(greet("You")) # Hello, You

The real power shows up when you combine them — a function that contains a loop, called from another loop. That's how grown-up programs are built. The widget below lets you watch exactly that happen.

Section 5 · Debugging + The Lifecycle

When (not if) something breaks: the programming lifecycle

One of the most important things you learned in Week 26 wasn't a syntax rule — it was a mindset. Real programs almost never work the first time. Debugging isn't a sign of failure; it's just part of the job. Every professional programmer in the world spends a huge chunk of their day tracking down bugs.

The good news: bugs come in three big flavors, and once you can name the flavor, you usually know how to fix it.

BUG TYPE 1 · ⛔
Syntax Error
Python can't even read your code. You see a red error before anything runs. Usually a typo, a missing colon, mixed-up = vs ==, or wrong indentation. The error message tells you the line.
BUG TYPE 2 · 💥
Runtime Error
Python starts running, then crashes mid-program. Common causes: dividing by zero, using a variable that doesn't exist yet, or trying to add a string and a number. Read the traceback — it points to the crash line.
BUG TYPE 3 · 🤔
Logic Error
The trickiest. The program runs all the way to the end with no error — but the answer is WRONG. Python can't help you here; you have to re-read your algorithm. Did you compare with > when you meant >=?
FIX MOVE · 🛠️
Print to Investigate
When stuck, drop in print() statements that show the value of variables at key points. You're not "fixing" yet — you're looking. Most bugs reveal themselves once you can see what your variables actually hold.

Debugging plugs into a bigger pattern called the programming lifecycle. It's not a straight line; it's a loop you walk every time you build something:

🌎 Real-World Connection

The first computer "bug" was a real bug. In 1947, a moth got trapped in a giant computer at Harvard called the Mark II — and stopped it from working. Engineers taped the moth into the logbook and wrote "first actual case of bug being found." From that moment on, "debugging" became programmer lingo. (The logbook is still on display at the Smithsonian.)

✅ Check for Understanding

Match each scenario to the bug type:

  1. Your code has if x = 5 and Python refuses to run it.
  2. Your program runs fine, but tells you that the average of 80, 90, and 100 is 30.
  3. Halfway through, your program crashes with "ZeroDivisionError."

Answers: 1 — syntax error, 2 — logic error, 3 — runtime error.

Putting it all together: that arc from Idea → Algorithm → Code, plus the Plan → Write → Test → Fix → Repeat lifecycle on top, IS Unit 3. Everything else — the syntax of for, the rules of indentation, the difference between = and == — are details that fit inside this big shape. Once you can hold the shape in your head, the details click into place.

📚 Key Vocabulary — Unit 3 Review

Computational thinking
A way of solving problems by decomposing them, finding patterns, abstracting essentials, and writing clear steps.
Algorithm
A finite, ordered list of steps that solves a problem when followed exactly. The bridge between idea and code.
Flowchart
A picture of an algorithm using ovals, rectangles, diamonds, and parallelograms connected by arrows.
Variable
A labeled box that holds a value. Set with =; you can read or change it as the program runs.
Conditional
An if/elif/else structure that lets a program choose between paths. Only one branch runs.
Loop
A way to repeat a block of code. for loops repeat a known number of times; while loops repeat until a condition becomes false.
Function
A named, reusable block of code. Define once with def, then call it as many times as you want with different inputs.
Debugging
The process of finding and fixing errors in code. Bugs come in three flavors: syntax, runtime, and logic.
