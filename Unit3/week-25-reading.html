Functions & Modular Programming

If you ever find yourself copy-pasting the same chunk of code over and over, your computer is trying to tell you something: stop. There is a smarter way to write programs — and its name is the function. This week, you'll learn how to package up a job once and use it forever.

Learning Objectives

By the end of this week, you will be able to:

1
Define a function in Python using the def keyword, give it a name, and write its body.
2
Pass data into a function using parameters and arguments — and tell the difference between the two.
3
Get data out of a function with a return statement, and explain why print and return are not the same.
4
Apply the DRY principle ("Don't Repeat Yourself") to break a long program into reusable, modular pieces.
01
Section 1 · Why Functions? — Don't Repeat Yourself

If you've written it twice, you've written it once too many

Imagine your teacher asks you to greet three new students by name. You sit down and type:

📝 The "copy-paste" way print("Hello, Maya! Welcome to class.") print("Hello, Maya! It's great to have you.") print("Hello, Lin! Welcome to class.") print("Hello, Lin! It's great to have you.") print("Hello, Theo! Welcome to class.") print("Hello, Theo! It's great to have you.")

It works! It also stinks. Look at what just happened: the same two sentences appeared three times, with only the name swapped. If your teacher tomorrow asks for thirty greetings instead of three, you'd be copy-pasting until the bell rings. And if you ever decide to change the wording — say, to "Welcome aboard" instead of "Welcome to class" — you'd have to find and edit it in every single place.

Programmers learned this lesson the painful way decades ago, and they wrapped the lesson into a single rule: DRY — Don't Repeat Yourself. If a piece of logic shows up more than once, package it up so you can use it everywhere with a single name.

That package is called a function. A function is a named, reusable block of code that does one specific job. You write the steps once, and from then on, every time you need that job done, you just say its name and the computer runs the steps for you.

📘 Vocabulary

Function — a named block of code that does one job and can be re-used anywhere in your program.

DRY — short for "Don't Repeat Yourself." A core programmer rule: never write the same logic twice. Bundle it into a function instead.

Reusable — code you can run again and again with different inputs, without rewriting it.

🌎 Real-World Connection

A microwave's "popcorn" button is a function. Somebody, once, figured out the perfect time and power level for popcorn — and then bundled all those settings under one button. Now you push it without thinking. You don't care how it works inside; you just trust that pressing the button gets you popcorn.

Functions in programming work the same way. The hard thinking happens once, when the function is written. After that, anyone can use it — including the future you who has forgotten the details.

Section 2 · Defining a Function

The four parts of every Python function

Writing a function is called defining it. In Python, you do that with one little keyword: def. Think of def as standing for "definition." It tells Python: "Hey, I'm about to teach you a new word — pay attention."

Here's the simplest possible function — one that says hello.

🧩 Anatomy of a function def say_hello(): print("Hello, world!")

Just two lines, but each piece matters. Let's slow down and look at the four parts of a Python function definition:

1
The def keyword
This is Python's signal that you are defining something new. Every function definition starts with def, no exceptions.
2
The function name
In our example, that's say_hello. Pick a clear, lowercase name with underscores between words. Names like x or do_thing are bad; say_hello, add_two_numbers, and is_even are good.
3
The parentheses ( ) and the colon :
The parentheses are where you'll list anything the function needs as input (we'll get to that next section). For now they're empty. The colon at the end says: "the body is about to start."
4
The body — indented!
Every line that belongs to the function is pushed to the right by 4 spaces (an "indent"). Python uses indentation to know what's inside the function and what's outside. If you forget the indent, your function won't work.

Here's the same function with the four parts labeled inline. Notice how the body sits underneath the colon, indented:

🔧 The four parts in one picture ┌── 1. keyword │ ┌── 2. function name │ │ ┌── 3. parentheses + colon ▼ ▼ ▼ def say_hello(): print("Hello, world!") <── 4. body (indented)

One critical thing to notice: defining a function is NOT the same as running it. When Python sees def say_hello():, it just memorizes the recipe — it does not cook the meal. Nothing prints to the screen yet. The function is sitting there, ready, waiting for someone to call it. To actually run it, you write the function's name followed by parentheses:

▶ Define vs. Call # Step 1 — DEFINE the function (Python memorizes it) def say_hello(): print("Hello, world!") # At this point, NOTHING has printed yet. ⚠️ # Step 2 — CALL the function (Python runs it) say_hello() # prints: Hello, world! say_hello() # prints it again say_hello() # and again
💭 Think About It

A recipe in a cookbook is a definition. The cake is what you get when somebody actually uses the recipe — that's the call. Writing def is like opening the cookbook and writing a new page. Calling the function is like turning to that page and following the steps. One book, many cakes.

💡 Did You Know?

The word function in programming was borrowed from math. In math class, f(x) = x + 2 is a function: it takes x in, does something to it, and gives a result back. Python's functions work exactly that way — only they can do far more than arithmetic. They can print messages, talk to the internet, draw pictures, even play sounds.

Section 3 · Parameters and Arguments

Passing data into the function

Our say_hello function works, but it's stuck. It always prints the same line, every time. That's not very useful — we wanted to greet different people: Maya, Lin, Theo, and whoever shows up next. The function needs a way to say: "Tell me whose name to use, and I'll do the rest."

The way you slide a piece of information into a function is with a parameter. A parameter is a placeholder name you list inside the parentheses when you define the function. It acts like a labeled box that gets filled in later, when someone calls the function.

📥 Adding a parameter def greet(name): # 'name' is the PARAMETER (the placeholder) print("Hello, " + name + "!") greet("Maya") # "Maya" is the ARGUMENT (the real value) greet("Lin") # "Lin" is the ARGUMENT this time greet("Theo") # "Theo" is the ARGUMENT this time

Run that, and you get three different greetings — from one function. That's the magic. The function is general; the data is specific.

Now let's get the vocabulary right, because students mix these up all the time:

📥 Parameter
The placeholder name
  • Lives inside the function's definition — the line with def.
  • Acts like a labeled empty box.
  • Doesn't have a real value yet — that comes from the caller.
  • Example: in def greet(name):, the parameter is name.
📦 Argument
The real value passed in
  • Lives inside the function call — the line that uses the function.
  • Is a real, specific value: a string, a number, a list…
  • Gets copied into the parameter when the function runs.
  • Example: in greet("Maya"), the argument is "Maya".

A way to remember: Parameter = Placeholder. Argument = the Actual value you pass in. The placeholder lives in the recipe; the actual value lives in the meal you're cooking right now.

Functions can take more than one parameter. Just separate them with commas:

📦 Two parameters, two arguments def add(a, b): # two parameters: a and b print(a + b) add(3, 4) # two arguments: 3 and 4 → prints 7 add(10, 90) # prints 100 add("Hello, ", "world!") # works with strings too — prints "Hello, world!"

Order matters! When Python sees add(3, 4), it puts 3 into a and 4 into b. If you swap them and write add(4, 3), the values swap too. For addition the answer is the same — but for something like subtraction, that's a totally different result.

✅ Check for Understanding

Spot the difference. In this code:

Which word is the parameter? Which is the argument? (Answers: parameter is message; argument is "go team".)

Section 4 · Return Values

Getting data out of the function

Parameters let us slide data into a function. But how do we get an answer back out?

Up to now, our functions have only used print. Printing is great when you just want a human to see something on the screen. But often, the rest of your program needs to do something with the answer — store it, math it, send it to another function. For that, you need return.

📤 The return statement def add(a, b): return a + b # hands the result back to whoever called us answer = add(3, 4) # the returned value (7) is stored in 'answer' print(answer) # prints 7 print(add(10, 90) * 2) # works inside expressions too — prints 200

The line return a + b says: "Calculate a + b, then hand that value back to whoever called this function." The function ends instantly when the return hits — no more lines after it will run.

This is the difference everyone has to figure out at some point:

🖨️ print
Shows it on the screen
  • Sends text to the human who's watching.
  • The function does not hand a value back.
  • Other code can't use the result.
  • Best when: a human needs to see something.
📤 return
Hands the value back
  • Gives the value to whoever called the function.
  • The result can be stored, printed, math'd, passed on.
  • Doesn't display anything by itself.
  • Best when: another part of the program needs the value.

You can absolutely do both — return a value and print it later — but the function itself should usually return, not print. That keeps it flexible. Watch:

⚡ Returns chain together def double(n): return n * 2 def add_one(n): return n + 1 result = double(add_one(4)) # add_one(4) → 5, then double(5) → 10 print(result) # prints 10

Because each function returns a value, you can plug one function's output straight into another's input. That is the secret of how big programs are built — small functions feeding each other.

Section 5 · Calling Functions Across Your Code

Modular programming — small parts, huge programs

Once you know how to define and call functions, something amazing becomes possible: you can stop thinking of your program as one giant wall of code, and start thinking of it as a team of small specialists. Each function is a worker that does one thing well. Your main code is the manager — it just calls each worker by name when needed.

This style is called modular programming, from the word module, meaning a self-contained piece. The opposite of modular is "spaghetti" — one tangled blob where everything is mixed up with everything else.

Suppose you're writing a game. The spaghetti way is one 800-line file where login, drawing, scoring, sound, and saving all run on top of each other. The modular way splits the same job into a handful of named functions:

🏗️ A modular game loop def login_user(username): # … checks the password and loads the profile return profile def draw_screen(player, score): # … paints the player and the score on screen def handle_keypress(player, key): # … moves the player when an arrow key is pressed def save_game(player, score): # … writes the score to a file # Main game loop — clean and readable! profile = login_user("avery") while playing: handle_keypress(profile, get_key()) draw_screen(profile, profile.score) save_game(profile, profile.score)

Even without knowing the inside of any function, you can read that main loop and understand what the game does. That is modular programming's superpower: you can think about the whole without drowning in the details.

One last idea before you wrap up: where do variables go? Each function has its own little workspace called its scope. Variables created inside a function exist only inside that function. They disappear when the function ends. That's a good thing — it means a variable named name inside greet() can't accidentally clobber a variable also named name somewhere else. Each function lives in its own bubble.

🫧 Scope in action def greet(name): message = "Hi, " + name # 'message' lives ONLY inside greet print(message) greet("Maya") print(message) # ❌ NameError — message doesn't exist out here!
💭 Think About It

Imagine if every grocery store cashier shared the same drawer of cash. One mistake and the whole system crashes. Instead, each cashier has their own drawer — their own scope. They can do their job in peace, and the store as a whole keeps running. Functions work the same way: each one gets its own private workspace, and that's exactly why programs with lots of functions stay calm under pressure.

💡 Did You Know?

The Python standard library — the set of functions that come built in when you install Python — contains over 200,000 lines of code bundled into thousands of pre-written functions. Every time you write print() or len() or input(), you're standing on the shoulders of programmers who wrote those functions decades ago so you wouldn't have to.

That's the DRY principle at planet scale — and it's why Python is one of the most popular languages on Earth.

📚 Key Vocabulary — Week 25

Function
A named, reusable block of code that does one specific job. Defined once, called many times.
def
The Python keyword that starts a function definition. Tells Python: "I'm about to teach you a new word."
Parameter
A placeholder name listed inside the parentheses of a function definition. It receives a value when the function is called.
Argument
The actual value passed into a function when it is called. Arguments fill the parameter slots.
Return
A statement that hands a value back from a function to whoever called it. Ends the function immediately.
Scope
The "bubble" a variable lives in. Variables created inside a function exist only there and disappear when the function ends.
Modular programming
Building a program from many small, focused functions instead of one giant blob. Easier to read, fix, and reuse.
Function call
A line that runs a function — the function's name followed by parentheses and any arguments, e.g. greet("Maya").
