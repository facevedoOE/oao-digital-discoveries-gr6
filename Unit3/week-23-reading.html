Logic: Conditionals — Making Programs Choose

A program that always does the same thing is boring. A program that chooses what to do next based on the situation is alive. This week you'll meet conditionals — the tiny words IF, ELSE, and ELIF — and learn how they let your code react to the world.

Learning Objectives

By the end of this week, you will be able to:

1
Use the six comparison operators (==, !=, <, >, <=, >=) to build true/false expressions about numbers and text.
2
Write IF and IF/ELSE statements that pick between two possible paths through a program.
3
Extend IF/ELSE with ELIF to handle three or more branches without messy nested code.
4
Translate a flowchart with decision diamonds into pseudocode and predict which branch will run for a given input.
01
Section 1 · Decisions, Decisions

Programs that choose what to do next

Stop and notice your morning. Your alarm goes off and a tiny chain of decisions begins. Is it a school day? If yes, get out of bed. If no, go back to sleep. Is it raining? If yes, grab a jacket. If no, leave it. Did you eat breakfast? If yes, head to the bus. If no, grab a granola bar first.

You don't think of it this way, but every "if this, then that" is a tiny conditional. You're constantly choosing one path or another based on what's true right now. The whole reason your day flows smoothly is that your brain is silently checking conditions and picking branches all morning long.

Computers do exactly the same thing — except they need you to spell out the rules. A program with no conditionals is like a script: it does the exact same thing every time you run it, no matter what's going on. That's almost never what you want. A program with conditionals can react to inputs, to the time of day, to what the user just clicked, to the score in a game. That's what makes it feel smart.

That diamond shape in the middle is called a decision diamond, and it's where the program asks a yes-or-no question. Whichever way the answer comes out, the program follows that arrow. The two arrows lead to two different futures — and only one of them happens on any single run.

Conditionals are everywhere in software. Look at three places you use them every day:

🎮 Video games
If your health drops to zero, the game ends. If you collect 100 coins, you get an extra life. If the boss is below half HP, it switches attack patterns. Every single rule in a video game is a conditional.
☔ Weather apps
If the chance of rain is over 50%, the app shows an umbrella icon. If the temperature is below freezing, it shows a snowflake. If there's a tornado warning in your area, it screams an alert at your phone.
📱 Your phone's lock screen
If you typed the right passcode, unlock. If wrong, count it as a failed attempt. If you've failed five times in a row, lock the phone for one minute. Conditionals all the way down.
📘 Vocabulary

Conditional — a piece of code that asks a yes/no question and runs different instructions depending on the answer.

Branch — one of the possible paths a conditional can take. Every conditional has at least two branches.

Decision diamond — the diamond shape used in flowcharts to mark a place where the program has to choose.

02
Section 2 · Comparisons — ==, <, >, and Friends

The six little symbols that ask yes-or-no questions

Before a program can act on a condition, it has to ask one. The way you ask a yes/no question in code is with a comparison operator. Each operator takes two values, compares them, and produces one of two answers: true or false. That answer is called a boolean expression, after the mathematician George Boole.

There are six comparison operators you'll meet over and over. They look almost like math, with one important twist:

Operator Means Example Result
== equal to 5 == 5 true
!= not equal to 5 != 7 true
< less than 3 < 10 true
> greater than 9 > 9 false
<= less than or equal to 9 <= 9 true
>= greater than or equal to 2 >= 8 false

Three of those need a closer look — they trip up almost every new programmer.

First: == versus =. In math class, you've used = your whole life to mean "is equal to." But in code, = means something completely different — it assigns a value. Writing x = 5 means "put the number 5 into the box called x." If you want to ask "is x equal to 5?" you have to use the doubled-up version: x == 5. Mixing them up is the most common bug a new programmer writes. Even pro programmers slip on it.

⚠️ Common Bug

If you write IF x = 5 when you meant IF x == 5, your code will not ask a question — it will change x to 5 and then act like the answer is always "yes." Suddenly your character has 5 health forever and you can't figure out why.

The rule: a single = changes things. A double == checks things.

Second: <= and >=. The "or equal to" part matters. 9 < 9 is false (9 is not less than itself), but 9 <= 9 is true. When you're writing a rule like "10 points or more wins a prize," the right operator is >=, not > — otherwise a player with exactly 10 points goes home empty-handed.

Third: !=. The exclamation point in code often means "not." So != is "not equal to." It's incredibly useful for things like "keep playing as long as the player isn't dead" or "only show the warning if the username is not empty."

💭 Think About It

Imagine you're writing the code for a roller coaster height check. The sign says "You must be at least 48 inches tall."

Which operator should you use to test a rider's height — > or >=? What happens to a kid who is exactly 48 inches if you pick wrong?

03
Section 3 · IF and ELSE

The two-path fork in the road

Once you can ask a yes/no question, you need a way to act on the answer. That's what IF statements do. An IF statement says: "check this condition, and if it's true, run these instructions." If the condition is false, the program just skips those instructions and keeps going.

In pseudocode, a basic IF looks like this:

PSEUDOCODE — IF only // Check if the player won IF score >= 100 THEN PRINT "You win!" END IF

Read it like English. "If score is at least 100, print 'You win!'" If the score is 99, the PRINT line never runs and the program moves on like nothing happened.

That's a one-branch conditional. But often you want a different thing to happen when the answer is no. That's what ELSE is for. ELSE means "otherwise" — it gives you a second branch for the false case:

PSEUDOCODE — IF / ELSE // Decide whether to grab an umbrella IF chanceOfRain >= 50 THEN PRINT "Bring an umbrella." ELSE PRINT "Leave it at home." END IF

Now there are exactly two possible outcomes, and one of them must happen. There is no situation where neither line prints. That's the magic of IF/ELSE: it splits the program into two paths and guarantees the program takes exactly one.

🌎 Real-World Connection

The login screen on every app you've ever used is just a giant IF/ELSE. If the password matches, let the user in. Else, show the angry red error message. There's nothing in between — you're either logged in or you're not. The whole "is this person who they say they are" question collapses into one boolean expression.

04
Section 4 · ELIF — When Two Paths Aren't Enough

More than two roads? Use ELIF.

IF/ELSE handles two outcomes. But what if there are three? Or four? Or seven? Picture how grades work: an A is 90 or above, a B is 80–89, a C is 70–79, a D is 60–69, and anything lower is an F. That's five different paths — way more than IF/ELSE can do on its own.

The clean answer is ELIF — short for "else if." ELIF lets you stack up extra conditions between the IF and the ELSE. The program walks down the list from top to bottom, and the first condition that turns out true is the one that wins. Everything below it gets skipped.

PSEUDOCODE — IF / ELIF / ELSE // Turn a number grade into a letter grade IF grade >= 90 THEN PRINT "A" ELIF grade >= 80 THEN PRINT "B" ELIF grade >= 70 THEN PRINT "C" ELIF grade >= 60 THEN PRINT "D" ELSE PRINT "F" END IF

Trace through it with grade = 85. The first check (85 >= 90) is false, so the program moves on. The second (85 >= 80) is true, so it prints "B" — and immediately jumps past every line below. The C, D, and F branches never even get tested. That's important: order matters in an ELIF chain, because only the first match runs.

You might wonder: why not just use a bunch of separate IFs, one for each grade? You could. But it would be longer, slower, and dangerously buggy — because if your conditions overlap (say, "grade >= 60" and "grade >= 70" are both true for an 85), more than one IF would fire and you'd print "B" and "C" and "D" all at once. ELIF makes it impossible for two branches to run.

✅ With ELIF
Exactly one branch runs
  • Conditions are checked top to bottom
  • The first one that's true wins
  • The rest are skipped automatically
  • Easy to read — one block, one outcome
⚠️ With many separate IFs
Branches can stomp on each other
  • Every IF is checked, even after one is true
  • Multiple branches may fire on one input
  • Conditions must never overlap or you'll get bugs
  • Easy to forget a case and miss inputs entirely

Section 5 · From Flowchart to Code

A worked example — "Guess my number"

Here's how a real programmer goes from an idea to working pseudocode. Imagine you want to write a tiny game: the computer is thinking of the number 42, the player guesses, and the program responds with "too low," "too high," or "got it!"

You start with a flowchart. Each diamond is a place where the program has to choose. Each rectangle is something the program does (printing a message, in this case). Arrows show the path the program takes.

Now translate it. Each diamond becomes one IF or ELIF. Each rectangle becomes a PRINT line. The result:

PSEUDOCODE — Guess my number // secret is 42; guess comes from the player IF guess > 42 THEN PRINT "too high" ELIF guess < 42 THEN PRINT "too low" ELSE PRINT "got it!" END IF

Three diamonds in the picture, three branches in the code. Notice the third path doesn't need its own comparison — by the time the program reaches ELSE, the only remaining possibility is that guess equals 42. The ELSE catches everything the IFs above didn't.

💡 Did You Know?

The boolean values true and false are named after George Boole, a 19th-century English mathematician. He invented an entire algebra of logic that uses just those two values — no numbers in sight. At the time it was a curiosity. A century later, electrical engineers realized that on/off switches inside a computer behave exactly like Boole's true/false, and his system became the foundation of every digital chip you've ever used.

Every IF you write is, deep down, asking the chip a Boolean question — and the chip answers in the same two-value language Boole sketched out 175 years ago.

📚 Key Vocabulary — Week 23

Conditional
A piece of code that asks a yes/no question and runs different instructions depending on the answer.
If statement
The simplest conditional — runs a block of code only when its condition is true.
Else clause
The "otherwise" path attached to an IF — runs when the IF's condition is false.
Elif
Short for "else if." Adds extra branches between an IF and an ELSE so a program can choose among three or more outcomes.
Comparison operator
A symbol like ==, !=, <, >, <=, or >= that compares two values and returns true or false.
Boolean expression
Any expression that evaluates to one of two values — true or false. The thing an IF actually checks.
Branch
One of the possible paths a conditional can take. Every IF has at least two; ELIF chains have more.
Decision diamond
The diamond shape in a flowchart that marks where the program has to make a yes/no choice.
